Kiến trúc Microservices

Nếu AuthService bị lỗi, APIService vẫn chạy bình thường. Mỗi service có thể nâng cấp, deploy, scale độc lập mà không ảnh hưởng nhau. Đây là xu hướng của các hệ thống lớn hiện đại (Netflix, Amazon...).

Luồng 1
Đăng nhập bằng Username + Password
Luồng phổ biến nhất — xảy ra mỗi khi user gõ tài khoản mật khẩu và nhấn "Đăng nhập"
Ví dụ thực tế

Giống như bạn rút tiền ATM: bạn cắm thẻ (username), nhập PIN (password), máy ATM kiểm tra thẻ thật không, PIN đúng không, rồi mới cho bạn rút tiền. Hệ thống này làm tương tự nhưng thay vì thẻ ATM là JWT Token — một chuỗi ký tự mà bạn mang theo mỗi lần gọi API.

📋 Từng bước chi tiết — Đăng nhập thường
1
User nhập username + password và nhấn "Đăng nhập"
Trình duyệt chuẩn bị một request POST. Dữ liệu được gửi dưới dạng JSON trong phần body.
// Dữ liệu gửi đi (request body) POST /api/auth/login { "username": "nguyenvan", "password": "Abc@1234" }
2
API Gateway nhận request, kiểm tra route
Gateway đọc file ocelot.json, thấy /api/auth/login → chuyển tiếp sang AuthService ở port 5002.
💡 Tại sao cần Gateway? Client chỉ cần biết 1 địa chỉ duy nhất (localhost:5000), không cần biết Auth đang chạy ở đâu.
3
AuthService tìm user trong database
Truy vấn SQL: "tìm user có username = 'nguyenvan'". Nếu không có → trả lỗi 401 ngay.
// EF Core query var user = await _userRepo.GetByUsernameAsync("nguyenvan"); if (user == null) return Unauthorized("Sai tài khoản");
4
Kiểm tra mật khẩu — PBKDF2 Hashing
Đây là bước quan trọng nhất về bảo mật. Database không lưu mật khẩu gốc mà lưu bản mã hóa.
// Bước 1: Lấy salt từ database byte[] salt = user.Salt; // random bytes lúc đăng ký // Bước 2: Hash mật khẩu user vừa nhập với salt đó string inputHash = HashPassword("Abc@1234", salt); // Bước 3: So sánh với hash lưu trong DB if (inputHash != user.Password) return Unauthorized("Sai mật khẩu");
💡 Tại sao không lưu mật khẩu thẳng? Nếu hacker lấy được database, họ chỉ thấy chuỗi hash vô nghĩa, không biết mật khẩu thật. PBKDF2 với 100.000 vòng lặp khiến brute-force cực kỳ tốn thời gian.
5
Tạo JWT Token — "Thẻ ra vào"
Mật khẩu đúng → tạo token chứa thông tin user, ký bằng khóa bí mật, có hạn sử dụng 480 phút (8 tiếng).
// Token chứa 3 thông tin (claims) { "Name": "nguyenvan", // username "NameIdentifier": "42", // userId trong DB "Role": "Admin", // quyền "exp": 1234567890 // hết hạn lúc nào } // Ký bằng: HMAC-SHA256 + SecretKey
6
Trả token về cho frontend
Server gửi lại token + thông tin user.
{ "token": "eyJhbGciOiJIUzI1NiJ9.eyJ...", "userId": 42, "username": "nguyenvan", "role": "Admin" }
7
Frontend lưu token và chuyển trang
React lưu vào localStorage, cập nhật AuthContext, rồi chuyển sang /admin (nếu Admin) hoặc / (nếu User thường).
// Lưu vào localStorage localStorage.setItem('token', response.token); localStorage.setItem('user', JSON.stringify(user)); // Chuyển trang navigate(user.role === 'Admin' ? '/admin' : '/');
✅ Đăng nhập thành công! Từ giờ mọi request sẽ gửi kèm token này.
Luồng 2
Đăng nhập bằng Google
OAuth 2.0 — Cho phép Google xác thực thay cho hệ thống
Ví dụ thực tế

Giống như thay vì bạn tự chứng minh "tôi là Nguyễn Văn A" với tòa nhà, bạn nhờ Google làm chứng: "Tôi là Google, xác nhận người này có tài khoản Gmail hợp lệ". Tòa nhà tin Google, nên cho bạn vào.

📋 Từng bước chi tiết — Google OAuth
1
User click nút "Đăng nhập với Google"
Google Sign-In SDK (chạy trong trình duyệt) hiện popup chọn tài khoản Gmail.
2
Google cấp ID Token
Sau khi user chọn tài khoản, Google trả về một chuỗi ID Token — bằng chứng chứng minh "người này đang đăng nhập vào Gmail hợp lệ".
💡 ID Token là JWT do Google ký — chứa email, tên, ảnh đại diện của user.
3
Frontend gửi ID Token cho AuthService
POST /api/auth/google { "idToken": "eyJhbGciOiJSUzI1NiJ9.eyJzdW..." }
4
AuthService xác minh token với Google
Server gọi API của Google để kiểm tra token có thật không (chống giả mạo).
// Gọi Google để xác minh GET https://oauth2.googleapis.com/tokeninfo ?id_token=eyJhbGci... // Google trả về thông tin user { "sub": "1234567890", // Google user ID (duy nhất) "email": "user@gmail.com", "name": "Nguyen Van A", "email_verified": "true" }
5
Tìm hoặc tạo tài khoản trong hệ thống
AuthService kiểm tra database: user này đã từng đăng nhập Google chưa?
// Tìm theo GoogleId trước var user = FindByGoogleId("1234567890"); // Nếu chưa có → tìm theo email if (user == null) user = FindByEmail("user@gmail.com"); // Vẫn chưa có → tạo tài khoản mới tự động if (user == null) { user = CreateUser(username: "gg_user", googleId: "1234567890"); }
💡 Username tự động có dạng gg_{email_prefix}. User không cần đặt mật khẩu.
6
Tạo JWT và trả về giống đăng nhập thường
Từ bước này trở đi giống hệt Luồng 1 bước 5–7. Token JWT được dùng cho mọi API tiếp theo.
✅ Tài khoản Google được liên kết. Lần sau đăng nhập cũng dùng Google, không cần nhớ mật khẩu.
Luồng 3
Xác minh OTP qua Email
One-Time Password — Mã dùng một lần gửi qua Gmail
Ví dụ thực tế

Giống như ngân hàng gửi SMS OTP khi bạn chuyển khoản. Mã chỉ có giá trị 5 phút, chỉ dùng được 1 lần. Hệ thống này gửi qua Gmail SMTP thay vì SMS.

📋 Từng bước chi tiết — OTP
1
Frontend yêu cầu gửi OTP
POST /api/auth/send-otp { "email": "user@gmail.com" }
2
Server tạo mã OTP ngẫu nhiên và lưu tạm
Tạo mã 6 chữ số ngẫu nhiên, lưu vào bộ nhớ/cache kèm thời gian hết hạn 5 phút.
otp = Random(100000, 999999) // VD: 482931 otpStore[email] = { code: otp, expiry: now + 5min }
3
EmailService gửi email qua Gmail SMTP
// Cấu hình SMTP trong appsettings.json Host: smtp.gmail.com Port: 587 (TLS) From: noreply@lumiereperfume.com // Nội dung email Subject: "Mã xác nhận OTP" Body: "Mã của bạn là: 482931 (hết hạn sau 5 phút)"
4
User nhập OTP và gửi lên
POST /api/auth/verify-otp { "email": "user@gmail.com", "otp": "482931" }
5
Server kiểm tra OTP
// Kiểm tra 3 điều kiện 1. OTP có trong store không? 2. OTP có khớp không? 3. OTP còn trong thời hạn 5 phút không? if (pass all) → 200 OK "Xác minh thành công" else → 400 "OTP sai hoặc hết hạn"
⚠️ Sau khi xác minh xong, OTP bị xóa khỏi store. Không thể dùng lại lần 2.
Luồng 4
Gọi API khi đã đăng nhập
Mỗi request sau khi đăng nhập đều phải kèm JWT Token — đây là luồng xảy ra hàng trăm lần khi dùng app
Ví dụ thực tế

Sau khi bảo vệ tòa nhà cấp cho bạn thẻ từ, bạn dùng thẻ đó quẹt mỗi khi vào phòng. Nhân viên phòng không hỏi bạn là ai nữa — họ chỉ xem thẻ có hợp lệ không. JWT Token đóng vai trò đó.

📋 Từng bước chi tiết — Lấy danh sách sản phẩm
1
React component gọi API
// Trong component React const products = await productApi.getAll();
2
Axios Interceptor tự động thêm Token
Không cần lập trình viên tự thêm tay — interceptor chạy tự động trước mỗi request.
// Interceptor trong api.js (chạy tự động) axiosInstance.interceptors.request.use(config => { const token = localStorage.getItem('token'); if (token) { config.headers.Authorization = `Bearer ${token}`; } return config; });
3
Request được gửi đến API Gateway
GET /api/products?page=1&limit=10 Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
4
Gateway chuyển tiếp sang APIService :5001
Ocelot đọc config, thấy /api/products → chuyển sang port 5001, giữ nguyên header Authorization.
5
JWT Middleware kiểm tra token
ASP.NET Core tự động xác minh token trước khi vào Controller.
// Middleware kiểm tra tự động: ✓ Chữ ký có đúng SecretKey không? ✓ Token có hết hạn chưa? ✓ Issuer = "BaseCore"? ✓ Audience = "BaseCore.Clients"? // Nếu sai bất kỳ điều nào → 401 Unauthorized
⚠️ Nếu trả về 401 → Axios Response Interceptor sẽ tự động xóa token khỏi localStorage và chuyển về trang /login
6
Controller → Service → Repository → Database
Sau khi xác thực xong, request đi theo luồng nghiệp vụ chuẩn.
ProductsController.GetAll() → ProductService.GetAllProductsAsync() → ProductRepository.GetAllAsync() → SQL: SELECT * FROM Products WHERE IsDeleted = 0 ← soft delete filter tự động ORDER BY Id DESC
💡 Global Query Filter: EF Core tự động thêm WHERE IsDeleted = 0 vào mọi query. Lập trình viên không cần tự viết điều kiện này.
7
Dữ liệu trả về và React cập nhật giao diện
200 OK [ { "id": 1, "name": "iPhone 16", "price": 25000000 }, { "id": 2, "name": "Áo thun", "price": 150000 }, ... ]
✅ React setState → component re-render → danh sách sản phẩm hiện ra màn hình.
🛡️ Kiểm tra quyền (Role-Based Access Control)
1
Admin-only endpoint có decorator đặc biệt
// Chỉ Admin mới POST được [HttpPost] [Authorize(Roles = "Admin")] ← decorator này public IActionResult Create([FromBody] ProductCreateDto dto) { ... }
2
ASP.NET tự kiểm tra Role trong token
// Token của User thường { "Role": "User" } → POST /api/products → 403 Forbidden // Token của Admin { "Role": "Admin" } → POST /api/products → 201 Created ✓
⚠️ Frontend cũng ẩn các nút Admin — nhưng đây chỉ là UX. Bảo mật thật sự là ở server check role trong token.
Luồng 5
Tạo đơn hàng — Transaction Database
Luồng phức tạp nhất — phải đảm bảo tính toàn vẹn dữ liệu: không bao giờ bán hàng khi không có hàng
Ví dụ thực tế

Hãy tưởng tượng bạn đặt mua 2 cuốn sách tại nhà sách. Nhân viên phải: (1) kiểm tra kho còn hàng không, (2) giữ sách đó lại cho bạn, (3) in hóa đơn — tất cả trong cùng một lần. Không thể in hóa đơn xong mới kiểm tra kho — nếu hết hàng thì rắc rối. Database Transaction đảm bảo điều này: làm tất cả hoặc không làm gì.

📋 Từng bước chi tiết — Tạo đơn hàng
1
User nhấn "Đặt hàng" ở trang giỏ hàng
POST /api/orders Authorization: Bearer eyJ... { "items": [ { "productId": 5, "quantity": 2 }, { "productId": 12, "quantity": 1 } ], "shippingAddress": "123 Đường ABC, Q1, TP.HCM", "recipientName": "Nguyễn Văn A", "paymentMethod": "COD" }
2
Lấy userId từ JWT Token
Server đọc token, lấy userId — không tin dữ liệu từ client gửi lên.
// Lấy userId từ claim trong token (không từ body) var userId = int.Parse( User.FindFirst(ClaimTypes.NameIdentifier).Value );
💡 Nguyên tắc bảo mật: Server không tin bất kỳ dữ liệu nào từ client về danh tính. Chỉ tin token đã được ký.
3
Mở Transaction — "Khóa" tất cả lại
await using var transaction = await _db.BeginTransactionAsync(); // Từ đây mọi thay đổi DB chưa thật sự lưu // Chỉ lưu khi gọi CommitAsync()
4
Kiểm tra từng sản phẩm trong giỏ
foreach (var item in orderItems) { var product = await _productRepo.GetByIdAsync(item.ProductId); // Sản phẩm có tồn tại không? if (product == null) throw new Exception("Sản phẩm không tồn tại"); // Còn đủ hàng không? if (product.Stock < item.Quantity) throw new Exception($"Hết hàng: {product.Name}"); }
5
Snapshot giá và trừ tồn kho
Đây là bước then chốt: lưu giá tại thời điểm mua vào OrderDetail. Dù sau này sản phẩm tăng/giảm giá, đơn hàng cũ vẫn giữ đúng giá lúc mua.
foreach (var item in orderItems) { // Snapshot giá hiện tại item.UnitPrice = product.Price; ← lưu vào OrderDetail // Trừ tồn kho product.Stock -= item.Quantity; _db.Update(product); }
6
Tạo bản ghi Order + OrderDetails
// Tạo đơn hàng chính var order = new Order { UserId = userId, TotalAmount = items.Sum(i => i.UnitPrice * i.Quantity), Status = "Pending", PaymentStatus = "Unpaid", OrderDate = DateTime.Now }; await _db.Orders.AddAsync(order); // Tạo chi tiết từng sản phẩm await _db.OrderDetails.AddRangeAsync(orderDetails);
7
COMMIT — Lưu tất cả vào database
await _db.SaveChangesAsync(); await transaction.CommitAsync(); // ✅ Chỉ đến đây mới thật sự lưu vào DB
✅ Đơn hàng tạo thành công. Trả về 201 Created kèm order ID.
💥 Điều gì xảy ra nếu có lỗi? — ROLLBACK
!
Rollback — Hoàn tác toàn bộ
Nếu bất kỳ bước nào ném exception (hết hàng, lỗi mạng, lỗi DB...) → transaction tự động rollback → tất cả thay đổi bị hủy → tồn kho trở về như cũ.
try { // ... tất cả bước trên ... await transaction.CommitAsync(); } catch (Exception ex) { await transaction.RollbackAsync(); // Tồn kho được hoàn trả tự động return StatusCode(500, "Đặt hàng thất bại"); }
⚠️ Không có transaction: Có thể trừ kho xong nhưng lưu đơn hàng thất bại → hàng bị "mất" mà không có đơn. Transaction ngăn chặn tình huống này.
Luồng 6
Xóa sản phẩm — Soft Delete
Không xóa thật — chỉ đánh dấu "đã xóa" để bảo toàn lịch sử đơn hàng
Ví dụ thực tế

Hãy nghĩ đến hồ sơ nhân viên nghỉ việc. Công ty không đốt hồ sơ — họ chuyển vào ngăn "Đã nghỉ". Vẫn tra cứu được nếu cần, nhưng nhân viên đó không xuất hiện trong danh sách đang làm. Soft Delete hoạt động tương tự.

❌ Xóa thông thường (KHÔNG dùng)
DELETE FROM Products WHERE Id = 5; // VẤN ĐỀ: OrderDetail vẫn còn FK đến Product 5 -- ERROR: FOREIGN KEY VIOLATION -- Hoặc xóa được nhưng mất lịch sử đơn hàng!
✅ Soft Delete (đang dùng)
UPDATE Products SET IsDeleted = 1 WHERE Id = 5; // ✓ Không vi phạm FK // ✓ OrderDetail vẫn xem được giá cũ // ✓ Có thể khôi phục nếu cần
🔍 Global Query Filter — Tự động lọc
1
Cấu hình một lần trong DbContext
// MySqlDbContext.cs — cấu hình 1 lần duy nhất modelBuilder.Entity<Product>() .HasQueryFilter(p => !p.IsDeleted);
2
Mọi query sau đó tự động lọc
// Code viết vậy _db.Products.ToList() // EF Core tự động thêm WHERE SELECT * FROM Products WHERE IsDeleted = 0 // Sản phẩm "đã xóa" không bao giờ xuất hiện!
✅ Lập trình viên không bao giờ cần nhớ thêm điều kiện IsDeleted. Tự động 100%.
Luồng 7
Phân quyền trang — Protected Routes
Ngăn User thường truy cập trang Admin trong React
Ví dụ thực tế

Trong tòa nhà, thẻ từ nhân viên bình thường không mở được cửa phòng Giám đốc. Dù biết số phòng, bạn vẫn không vào được. ProtectedRoute là bảo vệ đứng trước mỗi trang Admin.

📋 Luồng kiểm tra quyền khi truy cập /admin
1
User gõ URL /admin/products vào trình duyệt
2
React Router gặp ProtectedRoute
// Trong App.jsx — mọi route /admin đều qua đây <Route path="/admin" element={<ProtectedRoute />}> <Route path="products" element={<Products />} /> <Route path="users" element={<Users />} /> ... </Route>
3
ProtectedRoute kiểm tra 3 điều kiện
// ProtectedRoute.jsx const { user, isAdmin } = useAuth(); // 1. Có đăng nhập không? if (!user) return <Navigate to="/login" />; // 2. Có phải Admin không? if (!isAdmin()) return <Navigate to="/" />; // 3. Đủ điều kiện → cho vào return <Outlet />;
4
Kết quả
// Chưa đăng nhập → Redirect /login // Đăng nhập nhưng là User thường → Redirect / (trang chủ) // Là Admin → Hiển thị trang /admin/products ✓
⚠️ Đây chỉ là bảo vệ ở frontend. Hacker có thể bypass. Bảo mật thật sự là server kiểm tra [Authorize(Roles = "Admin")].
